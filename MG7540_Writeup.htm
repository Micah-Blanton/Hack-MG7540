Introduction
Why?

 

This year, I attended Microsofts BlueHat cybersecurity conference and met some great people in the industry. The event taught me a lot about the movements of big tech amongst emerging platforms.
It was both educational and inspirational, but what really blew me away was the explanation of Hardware and Firmware exploitation by reputable hackers Sick Codes and Nate Warfield, respectively.

 

Immediately after returning home, I binged tutorials from Defcon and similar conferences and began scrambling for hardware to break at local thrift shops. Today, we will be exploring the Motorola MG7540 and the wonderful methods of lockdown seemingly imposed by Broadcom.

The first step to any hacking project is identifying what exactly you are working with. Think of this as the uname -a step, if you come from a pentesting background. The difference here is that we can collect important information about the device long before we ever break into it. We know what were working with is a cable modem/router combo from the name. According to the talks at BlueHat and Defcon, any device that transmits data wirelessly must be registered with the FCC; therefore, we start by copying the FCC ID from the image above and pasting it into their search box under the Grantee Code (first 3-5 characters) and Product Code (remaining characters) sections. In this example, it would look like this:

Grantee Code: 2AF5P
Product Code: MG7540

The search returns a list of products that are roughly the same. I like to choose one of the older options, in case applications to remove information have been processed. Click Detail on whichever you choose to see the list of PDFs containing all the products information. In this case, Internal Photos.PDF shows some very interesting shots of the modems board. When opening the modem, there will be two sections shielded with metal. These sections usually protect components the manufacturer wants to restrict access to.

 

At this point, we can choose to acquire the modem and open it up, or abandon it and choose a different device for our project. I chose to buy the modem and open it up. It was only $5 at Goodwill and the 4 contacts exposed in the image above have me intrigued. Usually, contacts like these, in lines of 3 or 4 are debugging interfaces that can give us access to the device. In many cases, these interfaces use the UART protocol, which we can decode by soldering some pins to the contacts and hooking them up to an inexpensive Saleae style logic analyzer. Before we do any interfacing with the device, lets inspect the rest of the PCB, in case there are other vulnerable spots.

 

One might wonder what the IC shown above might be and whether it is useful. I looked it up and it is a type of volatile memory from Micron. By volatile, I mean that the data stored on the chip is erased with every power cycle of the device, so reading from it would not return much useful information, unless the device was powered on. However, we cannot read from it while the device is on because it is in BGA (Ball Grid Array) format, meaning all of the leads we would want to connect to are on the bottom of the IC and inaccessible without using a heat gun to completely remove the IC from the PCB. That said, the memory will still be useful later depending on how the debugging interface is connected to the system.





 

 

Upon further inspection, I found an 8-legged IC on the bottom of the PCB. 8 legs often means storage. There were a lot of false positives with 8-legged transformers and transistors and whatnot, but after looking up the product code printed on it, I found that this one was a Winbond SPI Flash memory IC. I quickly ordered a SOP8 SOIC clip to easily connect to the IC without having to solder anything. The clip came bundled with a CH341a flash reader/writer, so I used it to dump a binary from the chip with a tool called Neoprogrammer (formerly AsProgrammer). Using highly recommended tool called Binwalk to analyze the binary, I was not able to extract the filesystem. I could read a few lines in plaintext using the strings command in Kali Linux, which indicated that the 7z file Binwalk found, but could not extract, was a false positive, as properly compressed files would not return any strings in plaintext.

 

 

What now?

 

I am currently investigating the apparent UART interface on the device and have put the binary analysis on hold. Future updates are to come.
